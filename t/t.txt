

1) ВИ «Регистрация в системе»

Цель
Создание нового аккаунта для доступа к полному функционалу.

Акторы
Пользователь (Гость).

Предусловие
Пользователь не авторизован.

Основной успешный сценарий

Пользователь нажимает кнопку «Регистрация» на главной странице или форме входа.

Система отображает форму регистрации с выбором роли: «Студент» или «Волонтер».

Пользователь выбирает роль.

Система адаптирует поля формы под выбранную роль (скрывает/показывает специфичные поля).

Пользователь заполняет обязательные поля: ФИО, Email, Пароль, Возраст, Пол + (Факультет/Направление/Курс для студента ИЛИ «О себе» для волонтера).

Пользователь соглашается с правилами обработки персональных данных и нажимает «Зарегистрироваться».

Система проверяет уникальность Email и корректность формата введенных данных.

Система создает учетную запись со статусом «Не подтвержден».

Система отправляет ссылку для подтверждения аккаунта на указанный Email.

Система отображает сообщение: «Ссылка для подтверждения отправлена на почту».

Альтернативный сценарий №1: Email уже занят
Предусловие: На шаге 7 система обнаруживает, что Email уже есть в БД.
7a. Система подсвечивает поле Email красным и выводит сообщение: «Пользователь с таким Email уже существует».
7b. Пользователь вводит другой Email или переходит к форме восстановления пароля.
7c. Сценарий возвращается к шагу 6.

2) ВИ «Авторизация пользователя»

Цель
Вход в систему для доступа к личному кабинету.

Акторы
Пользователь.

Предусловие
Пользователь имеет зарегистрированный аккаунт.

Основной успешный сценарий

Пользователь открывает форму авторизации.

Пользователь вводит Email и Пароль.

Пользователь нажимает кнопку «Войти».

Система проверяет существование пользователя в базе данных.

Система сверяет хеш введенного пароля с сохраненным в базе.

Система проверяет, подтвержден ли Email (если это требование системы).

Система создает сессию пользователя и перенаправляет его в Личный кабинет (или на страницу, с которой был переход).

Альтернативный сценарий №1: Неверный пароль
Предусловие: На шаге 5 пароль не совпадает.
5a. Система выводит сообщение: «Неверный логин или пароль».
5b. Система предлагает ссылку «Забыли пароль?».
5c. Пользователь повторяет ввод данных (возврат к шагу 2).

Альтернативный сценарий №2: Восстановление пароля
Предусловие: Пользователь забыл пароль и нажал «Забыли пароль?» на форме входа.

Пользователь вводит Email, привязанный к аккаунту.

Система отправляет проверочный код (или ссылку) на почту.

Пользователь вводит полученный код в форму на сайте.

Система проверяет код.

Пользователь задает новый пароль и подтверждает его.

Система сохраняет новый пароль и перенаправляет на форму авторизации (возврат к шагу 1 основного сценария).

3) ВИ «Просмотр списка мероприятий»

Цель
Ознакомиться с доступными активностями и выбрать интересные.

Акторы
Пользователь (авторизованный или гость).

Основной успешный сценарий

Пользователь переходит в раздел «Мероприятия».

Система загружает список мероприятий из базы данных, сортируя их по дате (от ближайших к дальним).

Система отображает карточки мероприятий (Название, Дата, Место, Краткое описание, Статус регистрации).

Пользователь применяет фильтр «Только активные» (или выбирает категорию, например, «Спорт», «Наука»).

Система обновляет список, скрывая прошедшие или неактуальные события.

Пользователь нажимает на карточку интересного мероприятия для просмотра деталей.

Альтернативный сценарий №1: Список пуст
Предусловие: В базе нет мероприятий, удовлетворяющих фильтру.
5a. Система отображает сообщение: «Мероприятия по заданным параметрам не найдены».
5b. Система предлагает кнопку «Сбросить фильтры».

4) ВИ «Поиск мероприятия»

Цель
Найти конкретное событие по ключевым словам.

Акторы
Пользователь.

Предусловие
Пользователь находится в разделе «Мероприятия».

Основной успешный сценарий

Пользователь активирует поле поиска.

Пользователь вводит поисковый запрос (например, «Хакатон» или «Волейбол»).

Пользователь нажимает кнопку «Найти» (или поиск срабатывает динамически при вводе).

Система ищет совпадения в названиях и описаниях мероприятий.

Система отображает список найденных мероприятий.

Пользователь выбирает нужное мероприятие из результатов поиска.

Альтернативный сценарий №1: Ничего не найдено
Предусловие: На шаге 4 совпадений не найдено.
5a. Система выводит сообщение: «По запросу "[текст]" ничего не найдено».
5b. Пользователь корректирует запрос (возврат к шагу 2).

5) ВИ «Подача заявки»

Цель
Зарегистрироваться участником на выбранное мероприятие.

Акторы
Пользователь (авторизованный).

Предусловие
Пользователь авторизован и открыл страницу конкретного мероприятия. Регистрация на мероприятие открыта.

Основной успешный сценарий

Пользователь нажимает кнопку «Подать заявку» на странице мероприятия.

Система проверяет, не подавал ли пользователь заявку ранее.

Система открывает форму заявки (автоматически подставляя ФИО и контакты из профиля, если нужно).

Пользователь заполняет дополнительные поля, специфичные для этого мероприятия (например, «Размер футболки» или «Наличие ноутбука»).

Пользователь нажимает кнопку «Отправить».

Система сохраняет заявку в базе данных со статусом «На рассмотрении» (или «Ожидание»).

Система отображает сообщение «Заявка успешно отправлена».

Кнопка «Подать заявку» меняется на «Заявка подана».

Альтернативный сценарий №1: Дубликат заявки
Предусловие: На шаге 2 система видит существующую заявку.
2a. Система уведомляет: «Вы уже подали заявку на это мероприятие».
2b. Система предлагает перейти к просмотру заявки в Личном кабинете.

Альтернативный сценарий №2: Мест нет
Предусловие: Лимит участников исчерпан.
1a. Кнопка «Подать заявку» неактивна или заменена на «Мест нет».
1b. Пользователь не может инициировать сценарий подачи.

6) ВИ «Просмотр статуса заявки»

Цель
Узнать решение организаторов по поводу участия.

Акторы
Пользователь.

Предусловие
Пользователь авторизован.

Основной успешный сценарий

Пользователь переходит в раздел «Мои заявки» в Личном кабинете.

Система делает запрос к базе данных по ID пользователя.

Система отображает список всех заявок с названием мероприятия, датой и текущим статусом (цветовая индикация: желтый — ожидание, зеленый — одобрено, красный — отклонено).

Пользователь нажимает на конкретную заявку.

Система разворачивает детали заявки, включая комментарий администратора (если есть, например, причина отказа или инструкции для участников).

Альтернативный сценарий №1: Отмена заявки
Предусловие: Заявка в статусе «Ожидание», пользователь передумал.
4. Пользователь нажимает кнопку «Отменить заявку».
5. Система запрашивает подтверждение.
6. Система меняет статус на «Отменена пользователем» и освобождает место на мероприятии.

7) ВИ «Управление мероприятиями»

Цель
Создание нового события или редактирование существующего.

Акторы
Администратор.

Предусловие
Администратор авторизован и имеет права на управление контентом.

Основной успешный сценарий

Администратор переходит в панель управления, раздел «Мероприятия».

Администратор нажимает кнопку «Создать мероприятие».

Система открывает форму создания.

Администратор заполняет поля: Название, Описание, Дата/Время начала и конца, Место проведения, Лимит участников, Крайний срок регистрации, Тип мероприятия.

Администратор загружает обложку (изображение) мероприятия.

Администратор нажимает «Сохранить и опубликовать».

Система проверяет заполнение обязательных полей.

Система сохраняет мероприятие в БД и делает его видимым для пользователей.

Альтернативный сценарий №1: Ошибка валидации
Предусловие: Не заполнена дата мероприятия.
7a. Система подсвечивает поле «Дата» и блокирует сохранение.
7b. Администратор вносит исправления (возврат к шагу 4).

8) ВИ «Обработка заявок»

Цель
Модерация списка участников (одобрение или отклонение).

Акторы
Администратор.

Предусловие
Есть поступившие заявки в статусе «Ожидание».

Основной успешный сценарий

Администратор открывает список заявок конкретного мероприятия.

Система отображает таблицу с данными кандидатов.

Администратор просматривает анкету участника.

Администратор принимает решение и нажимает кнопку «Одобрить» или «Отклонить».

(Опционально) При отклонении Система запрашивает причину отказа.

Система обновляет статус заявки в базе данных.

Система инициирует отправку уведомления пользователю (вызов ВИ №10).

Счетчик доступных мест на мероприятии обновляется.

Альтернативный сценарий №1: Массовое одобрение

Администратор отмечает галочками несколько заявок.

Нажимает «Одобрить выбранные».

Система обновляет статусы всех выбранных заявок.

9) ВИ «Экспорт списка участников»

Цель
Получение списка участников для оффлайн-работы (чек-лист на входе, печать бейджей).

Акторы
Администратор.

Предусловие
У мероприятия есть одобренные участники.

Основной успешный сценарий

Администратор открывает страницу управления мероприятием.

Администратор нажимает кнопку «Экспорт участников».

Система предлагает выбор формата (Excel, PDF, CSV) и фильтр (например, «Только одобренные»).

Администратор выбирает параметры и нажимает «Скачать».

Система формирует файл с полями: ФИО, Группа/Курс, Контакты.

Браузер начинает загрузку файла на устройство администратора.

10) ВИ «Уведомления о статусе заявки»

Цель
Автоматическое информирование студента об изменении статуса его заявки.

Акторы
Система (таймер или триггер события).

Предусловие
Статус заявки пользователя был изменен Администратором (или автоматически).

Основной успешный сценарий

Система фиксирует изменение статуса заявки (триггер из ВИ №8).

Система генерирует текст письма/уведомления на основе шаблона (например: «Ваша заявка на [Название] одобрена!»).

Система отправляет письмо на Email пользователя (и/или Push-уведомление в профиль).

Система делает запись в логах об успешной отправке.

Альтернативный сценарий №1: Ошибка отправки
Предусловие: Почтовый сервис недоступен или email некорректен.
3a. Система получает ошибку доставки.
3b. Система помечает уведомление как «Не доставлено» для администратора.
